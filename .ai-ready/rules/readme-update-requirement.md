# README 更新记录维护规范

## 规则描述

每次迭代（包括功能开发、Bug 修复、部署调整等）完成代码修改后，**必须**同步更新 `README.md` 文件中的「更新记录」章节。

## 触发条件

当满足以下任一条件时，必须执行此规范：

1. **代码发生变更**：任何源文件（`src/`、`prisma/`、`scripts/` 等）被修改
2. **版本号变更**：`package.json` 中的 `version` 字段被更新
3. **功能上线**：任何用户可见的功能或修复被合并到 master 分支

## 执行步骤

### 1. 确认版本号和变更内容

在提交代码前，确认：
- 当前版本号（查看 `package.json` 的 `version` 字段）
- 本次变更的 commit hash（使用 `git log -1 --oneline`）
- 本次变更的简要描述（功能/修复/优化）

### 2. 定位 README 更新记录章节

打开 `README.md` 文件，找到 `## 更新记录` 章节。

### 3. 添加新的更新条目

在「更新记录」章节的**最顶部**（日期最新的的位置）添加新的条目，格式如下：

```markdown
### YYYY-MM-DD

- `版本号` 变更摘要。
- `commit-hash` 具体变更点 1。
- `commit-hash` 具体变更点 2。
- `commit-hash` 具体变更点 3。
```

**示例：**

```markdown
### 2026-04-16

- `0.1.54` 修复成员列表滚动问题并开放主题设置权限给所有成员。
- `0eaf770` 非房主成员列表改为 `flex-1 overflow-y-auto`，始终显示滚动条。
- `0eaf770` 房主设置区域添加 `max-h-[24rem] overflow-y-auto pr-1`。
- `0eaf770` 主题设置移到 isOwner 块外，所有成员都可以设置主题。
```

### 4. 提交 README 变更

将 `README.md` 的变更与其他代码变更一起提交：

```bash
git add README.md
git commit -m "fix: v0.1.54 描述你的变更"
```

或者如果已经提交，使用 `git commit --amend` 合并。

### 5. 推送到远程

```bash
git push origin master
```

## 格式要求

### 日期格式
- 使用 ISO 8601 格式：`YYYY-MM-DD`
- 示例：`2026-04-16`

### 版本号格式
- 使用语义化版本号，不带前缀
- 示例：`0.1.54`（**不是** `v0.1.54`）

### Commit Hash 格式
- 使用简短 hash（7 位）
- 用反引号包裹
- 示例：`0eaf770`

### 变更描述要求
- 第一行：概括性描述（包含版本号）
- 后续行：具体的技术变更点（包含 commit hash）
- 每条描述简洁明了，不超过 80 字符
- 使用中文描述

## 注意事项

1. **必须性**：这是强制要求，不是可选操作
2. **时效性**：必须在代码提交前或提交时同步更新，不可事后补写
3. **位置**：新条目必须放在「更新记录」章节的最顶部（日期最新）
4. **一致性**：README 中的版本号和 MAINTAIN.md 中的版本号必须一致
5. **完整性**：重要变更点都应该列出，不要遗漏

## 检查清单

在推送到远程前，请确认：

- [ ] `README.md` 已更新「更新记录」章节
- [ ] 新条目放在了更新记录的最顶部
- [ ] 日期格式正确（YYYY-MM-DD）
- [ ] 版本号正确（与 package.json 一致）
- [ ] Commit hash 正确
- [ ] 变更描述清晰准确
- [ ] README 已添加到 git 提交中

## 违规后果

如果不遵守此规范：
- 代码审查时会被要求补充
- 可能导致部署流程阻塞
- 影响版本追溯和历史查询

## 相关文件

- `README.md` - 项目文档，包含更新记录章节
- `MAINTAIN.md` - 维护笔记，更详细的技术变更说明
- `package.json` - 项目版本定义
